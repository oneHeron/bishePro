## 1. 第五章在答辩中的定位

第五章不是单纯展示页面，而是要说明：

- 前文研究内容已经工程化落地。
- 系统完成了“方法、数据集、指标、运行、结果”的统一集成。
- 平台具备可扩展、可运行、可追溯、可验证的应用价值。

答辩时建议把这一章讲成一句主线：

> 为了将前文社区检测研究内容转化为可实际使用的应用系统，本文设计并实现了 OpenCDP 社区检测集成平台，支持方法管理、数据集管理、指标评估、异步运行调度以及结果可视化展示。

## 2. PPT页数建议

建议本章使用 `6 页`，控制在 `4~6 分钟` 内讲完。

推荐顺序：

1. 系统建设目标与设计思路
2. 系统总体架构
3. 核心功能模块
4. 关键实现机制
5. 系统运行流程与界面展示
6. 系统效果与本章小结

## 3. 整体制作原则

### 3.1 每页都要有“结论”

标题不要只是“系统架构”，而要尽量写成结论式表达，例如：

- `OpenCDP实现了社区检测任务的一体化管理`
- `系统采用分层与插件化架构设计`
- `异步运行机制提升了任务执行与结果管理能力`

### 3.2 每页控制在 3 到 5 个点

- 每页只讲一个中心问题。
- 不要密密麻麻放大段文字。
- 图和结论优先，细节在口头补充。

### 3.3 本章截图建议

建议准备 4 张系统截图：

1. 首页或资源总览页
   - 路由：`/`
   - 作用：展示平台整体界面与资源浏览能力
2. 运行配置页
   - 路由：`/run`
   - 作用：展示“方法 + 数据集 + 指标 + 参数”的任务配置
3. 结果详情页
   - 路由：`/result/{id}`
   - 作用：展示指标、日志、状态、可视化结果
4. 运行历史页
   - 路由：`/history`
   - 作用：展示任务追踪、状态流转与历史管理

## 第2页：系统总体架构

### 页面标题

`系统采用分层与插件化相结合的总体架构`

### 页面文字

- 前端展示层：负责资源浏览、任务配置、结果展示与历史管理。
- 后端接口层：基于 FastAPI 提供公开资源、认证与运行管理接口。
- 中间运行层：负责异步调度、任务执行与取消控制。
- 数据持久化层：保存用户信息、运行记录、日志、指标结果与可视化结果。

### 页脚总结语

`通过分层设计将界面交互、业务编排、算法执行和数据存储解耦，提升了系统的可维护性与可扩展性。`

### 口播建议

“在总体架构上，系统采用前后端分离和分层设计。前端负责交互展示，后端负责接口和业务逻辑，中间运行层负责异步任务调度，底层负责持久化存储。同时，方法、数据集和指标通过统一注册机制接入，保证了系统具备良好的扩展能力。”

## 第4页：关键实现机制

### 页面标题

`系统通过插件化、异步化和可追溯机制保障工程可落地性`

### 页面文字

#### 关键机制 1：插件化注册

- 方法、数据集、指标统一通过注册中心接入。
- 新增模块时无需修改核心运行主流程。

#### 关键机制 2：异步运行调度

- 提交任务后立即返回 `run_id`。
- 后端在异步线程或队列中执行任务。
- 支持 CPU / GPU 队列分流和远程 worker 队列。

#### 关键机制 3：运行记录落库

- 保存参数、随机种子、日志、耗时、指标结果、版本信息。
- 支持任务追踪与实验复现。

### 口播建议

“这一页想强调的是，第五章并不是简单实现了一个前端界面，而是在系统底层实现了三项关键能力。第一是插件化接入，第二是异步运行调度，第三是运行记录的可追溯保存。这三部分共同支撑了系统的工程化应用价值。”

### 配图建议

- 左边画“注册中心”
- 中间画“异步任务队列”
- 右边画“运行记录表”

## 第5页：系统运行流程与界面展示

### 页面标题

`系统完成了从任务配置到结果展示的完整闭环`

### 页面文字

#### 运行流程

1. 用户选择方法、数据集、指标并配置参数。
2. 前端提交任务，请求后端创建运行记录。
3. 运行器异步调度算法执行并计算评价指标。
4. 系统保存日志、状态、结果与可视化数据。
5. 前端轮询任务状态并展示最终结果。

### 口播建议

“从使用流程来看，系统支持用户完成从任务配置到结果查看的完整链路。提交任务后，后端首先进行校验并创建运行记录，然后调度方法执行，计算指标并回写结果。前端通过轮询机制获取状态变化，最终展示指标、日志和社区划分结果。”
