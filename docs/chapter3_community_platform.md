# 第3章 社区检测集成平台的设计与实现

## 3.1 章节概述

针对前文提出的社区检测方法研究需求，本文进一步构建了一个可扩展的社区检测集成平台，旨在将“算法实现、数据集管理、评价指标计算、异步运行调度、结果可视化与历史追踪”纳入统一系统。该平台面向两类用户场景：其一，未登录用户可浏览方法、数据集与指标信息；其二，登录用户可提交运行任务、跟踪执行状态并查看结果。

本章从系统需求、总体架构、关键模块、数据持久化设计、核心业务流程及测试验证六个方面，系统阐述平台的工程化实现过程。

## 3.2 需求分析与设计目标

### 3.2.1 功能性需求

结合社区检测实验流程，系统需满足以下核心功能：

1. 方法管理：支持内置与扩展社区检测方法的统一注册、展示与调用。
2. 数据集管理：支持本地数据集读取，并具备网络下载与缓存能力。
3. 指标评估：支持 NMI、ACC、ARI、模块度 Q 等常用指标的统一计算。
4. 任务运行：支持“方法 + 数据集 + 指标 + 参数”组合提交，并采用异步执行。
5. 结果呈现：展示状态、耗时、日志、指标值、社区划分结果与拓扑可视化。
6. 历史管理：记录并查询个人运行历史，支持删除、取消与复现实验参数。

### 3.2.2 非功能性需求

1. 可扩展性：方法、数据集、指标均通过注册机制接入，新增模块无需修改主流程。
2. 可维护性：后端采用分层架构，前后端接口契约清晰。
3. 可追溯性：每次运行记录参数、随机种子、版本信息、日志与结果。
4. 可用性：提供可直接操作的 Web 图形界面，降低复现实验门槛。

## 3.3 系统总体架构

### 3.3.1 技术路线

系统采用前后端分离架构：

1. 前端：Vue3 + Vite，实现资源浏览、运行配置、结果展示与历史管理。
2. 后端：FastAPI，提供认证、公共资源查询、任务提交与结果查询 API。
3. 算法与指标层：Python 插件化实现，统一注册到平台核心注册表。
4. 运行层：支持本地线程异步、RQ+Redis 队列异步两种执行后端。
5. 存储层：采用 SQLite（可替换）持久化用户与运行记录。

### 3.3.2 后端分层设计

后端严格按“API 层 / 服务层 / 运行器 / 插件 / 数据集 / 指标”组织：

1. API 层：`/public`、`/auth`、`/runs` 路由处理请求与响应。
2. 服务层：负责插件装载、数据集管理、业务校验等逻辑封装。
3. 运行器层：负责任务入队、异步执行、进程生命周期与取消控制。
4. 插件层：方法、数据集、指标通过统一注册接口接入。
5. 持久化层：ORM 模型定义用户与运行记录，支持状态流转与历史查询。

该分层将“业务编排”和“算法实现”解耦，避免在 API 中直接耦合具体方法代码，提高后续扩展效率。

## 3.4 关键模块实现

### 3.4.1 插件化注册机制

平台在启动阶段执行内置插件加载流程，统一注册方法、数据集与指标元信息。核心注册表维护三类映射：`methods`、`datasets`、`metrics`，并支持方法实例（`method_plugins`）直接调用。其优势在于：

1. 新增方法时仅需实现 `MethodPlugin` 接口并注册。
2. 新增数据集时仅需补充目录与目录项元数据。
3. 新增指标时仅需补充指标定义与计算函数。

在实现层面，平台已内置 Louvain、K-Means、NMF、DeepWalk 等基础方法，并扩展接入 Node2Vec、LPA、Infomap、DDGAE、CDBNE、CSEA 等方法，形成可持续扩展的算法池。

### 3.4.2 数据集管理与标准化加载

数据集模块采用“目录项 + 统一解析”的方式处理不同来源数据：

1. 本地数据集：如 Karate、Dolphins、Polbooks 等，直接读取 `edges.csv`、`labels.csv`（可选 `features.csv`）。
2. 可下载数据集：如 Cora、Citeseer，首次访问时自动下载压缩包并转换为平台标准 CSV。
3. 统一输出结构：向方法与指标层提供 `nodes`、`edges`、`labels`、`features` 四类输入。

该设计保证了“算法侧只关注标准输入”，不需要在每个方法中重复实现数据预处理。

### 3.4.3 异步运行与调度机制

平台支持三种运行后端模式：

1. `auto`：优先使用 RQ+Redis，失败后退化为本地线程异步。
2. `rq/redis`：强制使用队列执行。
3. `inline`：同步执行，主要用于测试。

任务提交后立即返回 `run_id`，后端异步完成以下操作：参数校验、数据加载、方法执行、指标计算、结果写回。系统支持 CPU/GPU 队列分流，并可依据任务参数路由到远程 worker 队列，满足高开销方法的资源调度需求。

此外，运行器实现了任务取消能力：当用户触发取消操作时，系统可标记取消请求并终止活跃子进程，确保长任务具备可控性。

### 3.4.4 评价指标计算模块

指标模块提供统一入口 `evaluate_metrics`，对 NMI、ACC、ARI、模块度 Q 进行集中计算，并内置错误提示机制：

1. 当指标依赖标签但数据集缺失标签时，返回可读性错误。
2. 当预测长度与节点数量不一致时，给出结构化异常信息。
3. 当图结构为空或不满足模块度计算条件时，明确失败原因。

该机制使实验结果不仅“可计算”，也“可解释、可定位问题”。

### 3.4.5 运行记录与结果可视化

每次运行均持久化以下关键字段：

1. 任务标识与用户信息：`run_id`、`user_id`。
2. 运行配置：方法、数据集、指标、参数、随机种子。
3. 执行过程：状态、日志、开始/结束时间、耗时。
4. 输出结果：指标结果、社区划分、错误信息、版本信息。

前端结果页在小规模网络场景下展示拓扑结构可视化（节点着色表示社区），并提供结果导出能力；在大规模网络场景下自动退化为摘要视图，以保障交互性能。

## 3.5 数据库与接口设计

### 3.5.1 数据库模型设计

系统核心数据表包括 `users` 与 `runs`：

1. `users`：存储用户名、密码哈希、创建时间。
2. `runs`：存储任务参数、状态流、日志、结果 JSON、耗时及时间戳。

其中，`runs` 表采用 JSON 字段保存可变结构（如 `params`、`results`、`metrics_result`、`version_info`），可有效兼容不同方法的参数异构性与结果差异。

### 3.5.2 接口契约与状态流转

系统 API 分为三类：

1. 公共资源接口：`/public/methods`、`/public/datasets`、`/public/metrics`。
2. 身份认证接口：`/auth/register`、`/auth/login`。
3. 运行管理接口：`/runs`（提交）、`/runs/{id}`（状态）、`/runs/{id}/results`（结果）、`/runs/me`（历史）。

任务状态遵循 `pending -> running -> finished/failed/cancelled` 的有限状态机。该状态机与前端轮询机制配合，可实现任务生命周期全程可观测。

## 3.6 前端交互实现

前端包含五类核心页面：

1. 首页：展示方法、数据集、指标概览与详情弹窗。
2. 登录注册页：完成用户认证并保存访问令牌。
3. 运行页：支持方法筛选、数据集选择、指标勾选、参数编辑与远程模式配置。
4. 结果页：展示运行状态、指标表、执行日志、社区划分及拓扑图。
5. 历史页：展示个人任务列表、进度、批量删除与详情跳转。

在交互策略上，系统将“可见反馈”作为优先原则，例如运行中状态提示、轮询刷新倒计时、失败原因展开显示、运行日志逐步呈现，从而提升实验操作的可理解性。

## 3.7 系统测试与验证

平台实现了基础自动化测试，覆盖以下关键链路：

1. 插件注册测试：验证方法、数据集、指标是否正确加载。
2. 数据集加载测试：验证本地数据预览与缺失文件错误提示。
3. 指标计算测试：验证理想划分与非理想划分下指标输出正确性。
4. API 集成测试：验证注册、登录、提交任务、查询结果完整流程。
5. 方法集成测试：验证典型方法在标准数据集上可完成“方法执行+指标计算”。

测试结果表明，平台已具备从资源浏览到异步运行、再到结果回显与历史追踪的端到端闭环能力，能够支撑后续算法对比实验与应用演示。

## 3.8 本章小结

本章围绕社区检测集成平台的工程实现，完成了从需求到架构、从模块到流程、从存储到验证的系统化说明。通过插件化设计与异步运行机制，平台在保证可扩展性的同时实现了较好的可维护性与可复现性。该平台不仅可作为本文实验部分的统一运行载体，也为后续接入新方法、新数据集和新评价指标提供了稳定基础。
